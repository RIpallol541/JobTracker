# JobTrack — полная инструкция по запуску

Проект поднимается через **Docker Compose**: три сервиса — **PostgreSQL** (БД), **backend** (FastAPI), **frontend** (собранный SPA за **Nginx**; запросы к `/api` проксируются на backend).

---

## 1. Требования

| Что нужно | Примечание |
|-----------|------------|
| **Docker** с поддержкой Compose | Windows/macOS: [Docker Desktop](https://www.docker.com/products/docker-desktop/). Linux: Docker Engine + плагин Compose. |
| Свободные порты | **80** (веб), **8000** (API напрямую), **5432** (PostgreSQL на хосте, если не занят). |
| Файл окружения | Копия `.env.example` → `.env` в **корне** репозитория (рядом с `docker-compose.yml`). |

Проверка, что демон Docker запущен:

```powershell
docker version
```

В выводе должны быть строки **Client** и **Server**. Если **Server** отсутствует — Docker Desktop не запущен или не установлен.

Создайте `.env` из шаблона:

**Windows:**

```powershell

```

**Linux / macOS:**

## 4. Сборка и запуск контейнеров

Из **корня** репозитория (где `docker-compose.yml`).

### Вариант А — в фоне (рекомендуется)

```powershell
docker compose up -d --build
```

- `--build` — пересобрать образы, если менялся код или Dockerfile.
- `-d` — контейнеры работают в фоне.

Первый запуск может занять **несколько минут**: установка зависимостей в образах (`pip`, `npm`), миграции Alembic, при `RUN_SEED=true` — загрузка демо-данных.

### Вариант Б — в текущем терминале (логи в консоли)

```powershell
docker compose up --build
```

Остановка: **Ctrl+C**, затем при необходимости:

```powershell
docker compose down
```

### Порядок старта

1. Контейнер **db** (PostgreSQL) — healthcheck `healthy`.
2. Контейнер **backend** — миграции, при включённом seed — скрипт демо-данных, затем Uvicorn; healthcheck `healthy`.
3. Контейнер **frontend** — Nginx со статикой.

Просмотр логов backend:

```powershell
docker compose logs -f backend
```

Статус сервисов:

```powershell
docker compose ps
```

## 5. Открытие приложения

| Ресурс | URL |
|--------|-----|
| **Веб-интерфейс** | http://localhost |
| **Swagger (документация API)** | http://localhost:8000/docs |
| **ReDoc** | http://localhost:8000/redoc |
| **Проверка работы API** | http://localhost:8000/health |

Запросы из браузера к API в обычном режиме идут на **тот же хост** (`http://localhost`): пути **`/api/...`** обрабатывает Nginx и перенаправляет на backend.

### Демо-учётная запись

Заполняется автоматически, если при старте backend выполнился seed (по умолчанию `RUN_SEED=true` в `.env` / `docker-compose.yml`) и пользователь ещё не существовал.

| Поле | Значение |
|------|----------|
| Электронная почта | `demo@example.com` |
| Пароль | `DemoPass123!` |

Можно также зарегистрировать нового пользователя на странице регистрации.

## 6. Остановка и данные базы

Остановить контейнеры:

```powershell
docker compose down
```

Данные PostgreSQL по умолчанию хранятся в **именованном Docker-томе** `postgres_data`. При **`docker compose down` без `-v`** том **сохраняется** — при следующем `up` база на месте.

Полностью удалить контейнеры **и** том с данными БД (осторожно, все данные пропадут):

```powershell
docker compose down -v
```

---

## 7. Что выполняется при старте backend

1. **`alembic upgrade head`** — применение миграций схемы БД.
2. Если **`RUN_SEED`** включён (значения вроде `true`, `1`, `yes`, `on`), запускается **`python -m scripts.seed`**.
3. Seed **идемпотентен**: если пользователь `demo@example.com` уже есть, повторная вставка демо-данных **не выполняется**.
4. Запуск **Uvicorn** на порту 8000 внутри контейнера (снаружи проброшен на **8000**).

## 8. Переменные окружения

Шаблон — файл **`.env.example`** в корне. Основные переменные:

| Переменная | Назначение |
|------------|------------|
| `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` | Учётные данные и имя базы в контейнере PostgreSQL |
| `JWT_SECRET` | Секрет подписи JWT (в продакшене — длинная случайная строка) |
| `JWT_ALGORITHM` | Алгоритм JWT (часто `HS256`) |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Время жизни access-токена |
| `CORS_ORIGINS` | Разрешённые для браузера origin, через запятую |
| `RUN_SEED` | Запускать ли демо-seed при старте backend |
| `LOG_LEVEL` | Уровень логирования backend |

Строка подключения **`DATABASE_URL`** для backend в Docker задаётся в **`docker-compose.yml`** (подстановка из `POSTGRES_*`), отдельно в `.env` для compose обычно не требуется.

## 9. Повторная заливка демо-данных (reseed)

Если демо уже создавалось, обычный seed при старте **ничего не добавит**. Чтобы **удалить** пользователя `demo@example.com` со всеми связанными данными и **заново** выполнить полный демо-набор (вакансии, отклики, собеседования, офферы, заметки):

1. Убедитесь, что в образ backend попали актуальные скрипты (после обновления кода из репозитория):

   ```powershell
   docker compose build backend
   docker compose up -d
   ```

2. Запустите скрипт:

   ```powershell
   docker compose exec backend python -m scripts.reseed_demo
   ```

   Альтернатива (тот же скрипт):

   ```powershell
   docker compose exec backend python /app/scripts/reseed_demo.py
   ```

Другие зарегистрированные пользователи **не удаляются**.

Если появляется ошибка **`No module named scripts.reseed_demo`**, пересоберите образ backend (`docker compose build backend`) и снова выполните команду `exec`.

## 10. Пересборка после изменений в коде

Полная пересборка всех образов:

```powershell
docker compose build --no-cache
docker compose up -d
```

Только backend или только frontend:

```powershell
docker compose build backend
docker compose build frontend
docker compose up -d
```

## 11. Устранение неполадок

| Симптом | Что сделать |
|---------|-------------|
| Ошибка `dockerDesktopLinuxEngine` / «pipe» / не найден сокет | Запустите **Docker Desktop** и дождитесь готовности. |
| В `docker compose ps` только **db** healthy, backend падает | `docker compose logs backend -f` — проверьте миграции, подключение к БД, наличие `JWT_SECRET`. |
| `No module named scripts.reseed_demo` | `docker compose build backend`, затем снова `exec` reseed. |
| Ошибки **bcrypt** при скриптах в контейнере | Пересоберите backend: `docker compose build --no-cache backend`. В проекте используется библиотека **bcrypt** напрямую. |
| `entrypoint.sh: no such file or directory` | Пересоберите backend без кэша: `docker compose build --no-cache backend`. В репозитории entrypoint вызывается через `/bin/sh` и обрабатывается CRLF. |
| Сборка backend падает на **таймауте pip** | В `backend/Dockerfile` заданы увеличенные timeout и retries. Повторите сборку; проверьте сеть, VPN, DNS. |
| В логах Postgres сообщения про **locale** | Для образа `postgres:alpine` часто **не мешают** работе. |

## 12. Локальный запуск без Docker (кратко)

Нужны установленные **Python 3.12**, **Node.js 20+**, **PostgreSQL**, файл настроек в духе `backend/.env.example` с корректным **`DATABASE_URL`**.

- **Backend:** каталог `backend` → виртуальное окружение → `pip install -r requirements.txt` → `alembic upgrade head` → `uvicorn app.main:app --reload --host 127.0.0.1 --port 8000` (переменная окружения **`PYTHONPATH`** должна указывать на каталог `backend`).
- **Frontend:** каталог `frontend` → `npm install` → `npm run dev` → интерфейс обычно на http://localhost:5173; в `vite.config.ts` настроен прокси **`/api`** на порт backend.

Подробности конфигурации — в `backend/.env.example` и `vite.config.ts`.
