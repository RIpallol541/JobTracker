# Описание проекта JobTrack для научного руководителя

Документ предназначен для предварительного ознакомления с выпускной квалификационной работой (или курсовым проектом): что решается, как устроено решение и какие технологии используются.

---

## 1. Тема и предметная область

**JobTrack** — веб-приложение для индивидуального учёта процесса поиска работы и **сравнительного анализа условий трудовых предложений (офферов)**.

Предметная область сочетает:

- **Учёт и планирование** — единое место для вакансий, откликов, этапов отбора, заметок и зафиксированных условий офферов (в отличие от разрозненных таблиц, чатов и почты).
- **Поддержка решения** — наглядное **сравнение офферов** по нескольким критериям (зарплата, бонусы, формат работы, отпуск, страховка, релокация и др.) с **настраиваемыми весами**, отражающими приоритеты соискателя.

Такой инструмент относится к классу персональных информационных систем и к задачам **многокритериального сравнения вариантов** в условиях неполной формализации предпочтений (веса задаёт пользователь).

---

## 2. Актуальность и цель работы

**Актуальность:** при одновременном ведении нескольких откликов теряется контекст (статусы, даты, условия); сравнение офферов «на глаз» не учитывает явно важность разных факторов. Специализированный сервис снижает когнитивную нагрузку и фиксирует историю принятия решений.

**Цель проекта:** разработать работоспособный **MVP (минимально жизнеспособный продукт)** — веб-сервис с регистрацией пользователя, хранением данных в СУБД, REST API и интерфейсом на стороне клиента, включающий **модуль сравнения офферов** с взвешенной агрегацией нормализованных показателей.

---

## 3. Задачи (что реализовано в рамках MVP)

1. **Модель предметной области:** пользователь, вакансии, отклики, собеседования, офферы, заметки; связи и ограничения целостности (в т.ч. один оффер на отклик в рамках модели данных).
2. **Безопасность:** хранение паролей в виде криптографических хэшей (bcrypt), аутентификация по **JWT**, разграничение данных по пользователю.
3. **Backend:** REST API (FastAPI), слой доступа к данным (репозитории), бизнес-логика (сервисы), миграции схемы БД (Alembic).
4. **Frontend:** одностраничное приложение (SPA), маршрутизация, работа с API, основные сценарии CRUD и дашборд.
5. **Сравнение офферов:** расчёт итогового балла по набору критериев с **настраиваемыми весами**, отображение разбивки по критериям и выделение лучшего варианта при текущих весах.
6. **Развёртывание:** контейнеризация (Docker Compose: PostgreSQL, backend, Nginx с фронтендом), воспроизводимый запуск и демонстрационное наполнение БД (seed).

## 4. Функциональность с точки зрения пользователя

| Модуль | Назначение |
|--------|------------|
| **Регистрация и вход** | Учётная запись, сеанс через JWT. |
| **Обзор (дашборд)** | Сводные показатели: количества сущностей, ближайшие собеседования, активные офферы, недавние заметки, упрощённые метрики «конверсии» между этапами. |
| **Вакансии** | Каталог отслеживаемых позиций (компания, ссылка, зарплатный коридор, формат работы и т.д.). |
| **Отклики** | Связь с вакансией, статус воронки, источник, даты, комментарии. |
| **Собеседования** | Привязка к отклику, этап, время, формат, результаты, заметки. |
| **Офферы** | Условия предложения (зарплата, бонусы, график, формат работы, льготы и др.), привязка к отклику. |
| **Сравнение офферов** | Настройка весов критериев, пересчёт рейтинга, таблица с вкладом критериев в итог. |
| **Заметки** | Общие и привязанные к вакансии, отклику или собеседованию. |

Демонстрационный вход и сценарии заполнения БД описаны в `README.md`.

## 5. Технологический стек

| Уровень | Технологии |
|---------|------------|
| Клиент | React 18, TypeScript, Vite, React Router, Axios, Tailwind CSS, Zustand |
| Сервер приложений | Python 3.12, FastAPI, Uvicorn |
| Данные | PostgreSQL 16, SQLAlchemy 2.x (синхронный режим), Alembic, драйвер psycopg 3 |
| Безопасность и контракты | Pydantic v2, JWT (python-jose), bcrypt |
| Инфраструктура | Docker Compose, Nginx (раздача статики и обратный прокси к API) |

Выбор стека ориентирован на распространённые в индустрии решения с явной документацией API (OpenAPI/Swagger из коробки у FastAPI) и возможностью показать работу на защите через Swagger и живой интерфейс.

---

## 6. Архитектура (кратко)

- **Клиент–сервер:** браузер обращается к фронтенду по HTTP; API запросы с того же origin идут на префикс `/api` и проксируются Nginx на backend.
- **Backend:** слоистая структура — маршруты (роутеры FastAPI) → зависимости (в т.ч. извлечение `user_id` из JWT) → сервисы → репозитории → модели SQLAlchemy. Ошибки приводятся к единому JSON-формату ответа.
- **Сравнение офферов:** выделенный сервис нормализует показатели по офферам пользователя и считает взвешенную сумму согласно переданным или дефолтным весам (логика может быть детализирована в пояснительной записке при необходимости).

---

## 7. Развёртывание и воспроизводимость

Проект запускается через **Docker Compose** (база данных, backend, frontend). Инструкции по установке зависимостей, переменным окружения и командам приведены в **`README.md`**; краткий чеклист — в **`docs/ЗАПУСК.md`**.

Это позволяет научному руководителю или комиссии **повторить запуск** на отдельной машине без ручной настройки СУБД и фронтенд-сборки, если установлен Docker.

## 8. Что можно развить в перспективе (за рамками текущего MVP)

- Экспорт отчётов (PDF/Excel), уведомления о дедлайнах.
- Углубление модели сравнения (нечёткие веса, сценарии «что если»).
- Мобильный клиент или PWA.
- Нагрузочное и интеграционное тестирование, CI/CD.

Перечень не является обязательством по объёму ВКР — приведён для контекста направлений развития.
